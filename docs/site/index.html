§1 立项动机

为什么做这个项目？为什么是短链？为什么这套技术栈？

短链服务（bit.ly 那种）在工程上有几个有意思的特性：读重写轻、热点极不均匀、Redis 命中率压倒一切、QPS 天花板靠系统调用而非业务逻辑。 这让它成为一个绝佳的"高并发后端 + 工程纪律"演练场。

为什么差异化做"大促级 + 自托管"

  • 商业短链太多（bit.ly / 阿里云 / 腾讯云）— 一定要差异化，否则讲不出故事
  • 玩具项目没说服力 — Day 1 就立"必须有 wrk 压测 + 故障演练 + 端到端对账"硬指标
  • 恰好命中两个工程加分项：高并发（前期性能调优）+ 全栈工程（中后期 OLAP/K8s/observability）

为什么选 PG 不选 MySQL

PG 16 支持声明式分区表（CREATE TABLE ... PARTITION BY RANGE）+ 部分索引（WHERE 谓词索引）。 click_events 表按 month 分区 + status='active' 部分索引，能省 90% 索引体积。MySQL 也能做但都要手动管，工程性差。

为什么号段双 buffer（而不是 Snowflake / UUID）

  • 顺序性：短码可读、可排序、Base62 反推 ID 调试方便
  • 双 buffer 预拉：本地 ring buffer 用到 80% 自动后台拉下一段，永远不阻塞主路径
  • UUID 短码丑且占空间；Snowflake 在 K8s 多副本下 worker_id 分配本身仍是中心化问题（v0.6 §8 决策稿详细对比）
核心理念 不追功能数量，追"每个版本能讲出来什么"。每个 v0.X 落地一组核心技术：v0.1 立项 / v0.2 性能 / v0.3 cache + 工程化 / v0.4 异步链路 / v0.5 OLAP / v0.6 K8s。

§3 v0.1 – v0.3 — 单机 baseline → fasthttp → L1 cache

从 21k RPS 起步到 86k RPS。3 个版本拉通"profile-first + 测量驱动"工程纪律

v0.1 单机 baseline（21k RPS）

Day 1-5 从 0 起一个真实可跑的短链服务。技术栈：fasthttp 前 net/http + pgxpool + go-redis + caarlos0/env。

  • 短码生成：号段双 buffer + Base62 + 位置混淆 (id × P) mod N 双射，mulModSafe 防 int64 溢出
  • POST /api/links：Stripe 风格幂等键 + DB unique 约束兜底 + SSRF 4 层防御（IP 解析后再校验，防 DNS rebind）
  • GET /:code：LinkCache.GetOrLoad 一站式 + 302（让客户端不缓存，统计才准）+ wrk 压测达 21k RPS
  • 瓶颈定位：pprof 显示 syscall 占 69%，单进程 net/http 的天花板

v0.2 探底 + fasthttp（24k RPS / +13% RPS / -29% CPU per req）

Day 6-7 用真实 profile 数据替代猜测，决定下一个优化方向。

Day 6 pprof 探底（5h）

  • top10: syscall 69%, runtime/sched 12%, alloc 4%
  • 结论：alloc 不是瓶颈，syscall 是真天花板
  • 优化方向：横向（多 process / 换栈底）而不是纵向（减 alloc）

Day 7 整层换 fasthttp（4h）

  • 零拷贝陷阱：ctx.UserValue("code").(string) 已经是 string，不要 cast 回 byte slice
  • +13% RPS / -29% CPU per req
  • 瓶颈平移到 Redis 网络往返（~500μs round-trip）

v0.3 L1 cache + 工程化（86k RPS / P99 24.7ms / L1 hit 99.97%）

Day 8-11 消除 Redis 网络瓶颈 + 把项目工程化到生产可用形态。

L1 进程内 LRU cache（Day 8）

  • 两层 cache：L1（hashicorp/golang-lru）+ L2（Redis）
  • nil-safe（缓存空值防穿透）+ TTL 分层（L1 短 1min / L2 长 10min）+ negative cache
  • +294% RPS / -78.6% alloc per req（每请求 alloc 从 ~4.5KB 降到 ~960B）

调参反向劣化的反直觉（Day 9）

反直觉发现 把 event buffer 加大反而 P99 升高了。原因：buffer 出口的 PG 单 flusher 是真瓶颈，buffer 大只是把丢失"延后"。 profile-first 的硬教训：先看瓶颈在哪，再调参。

Prometheus + Grafana 一键起栈（Day 10）

  • 闭包注入 vs interface：闭包注入避免 cache 包反向依赖 metric 包，零循环
  • normalizePath 控基数：把 /abc /xyz 这种动态短码都映射到 /:code label
  • 9% middleware 开销换永久可见性（86k RPS 含监控 vs 93k 裸跑）

安全 hardening 一次清完（Day 11）

code review 发现 6 个 HIGH 一次清完：

§4 v0.4 — Kafka 异步 + 故障域分离 ★

把 click 事件写库从主路径剥离到异步链路。整个项目最大跨越，故障域真分离的硬证据

背景：v0.3 留下的真问题

v0.3 的 86k RPS 是"主路径快"但异步出口慢：内存 buffer 满了就 drop event。在 wrk 60s 满载时实测 click 丢失率高达 49%。 修法不是加大 buffer（Day 9 已证伪），而是把出口换成 Kafka — 同步主路径 + 异步消费写库。

HLL UV（uniqHLL12）

Unique Visitor 不用精确 COUNT DISTINCT（要扫全列），用 HyperLogLog 近似算法： 

SELECT
    code,
    uniqHLL12(ip) AS uv_estimate  -- 0.81% 标准误，内存 -1~2 量级
FROM click_events_ch
WHERE ts >= now() - INTERVAL 1 HOUR
GROUP BY code
ORDER BY uv_estimate DESC
LIMIT 10;

★ DETACH MV 优化（Day 24-25，6.8× 消除）

发现 Day 24 故障演练时，SELECT * FROM system.processes WHERE current_database='slink_analytics' 发现 MV 写入路径占了 ClickHouse 51% CPU，远超预期。

根因：MV 的 fanout 路径（每个 INSERT 触发 SELECT trigger + INSERT target）。修法：DETACH MV → 直接 INSERT target → ATTACH MV。

优化前（ATTACH state）

  • 每个 Kafka 消费 batch 触发 MV 同步
  • MV CPU 占比：51%
  • endpoint P99：~10ms（数据写入路径 contention）

优化后（DETACH 后改写入 + 异步 ATTACH）

  • 批量直插 target，MV 异步追
  • MV CPU 占比：7.5%
  • endpoint P99：1.5 ms（目标 200ms 的 130×）
  • 6.8× CPU 消除
CH 运维红线（DETACH MV 教训沉淀） DETACH 期间任何崩溃 → ATTACH 失败 → 数据可能不一致。docs/architecture/v0.5-clickhouse.md §10 立了 CH 运维 SOP： DETACH 前先 backup metadata、ATTACH 后必跑端到端对账验证。

★ 真隐患：Kafka K8s advertised listener（Day 28 现场触发）

现象 fresh K8s Pod sent=6 acked=0 dropped=6 100% silent drop。v0.6 Phase 1 起异步路径从未工作过， 但 Phase 2 演练全 PASS（RPS / timeout / 5xx 都正常） — 因为 redirect 主路径不依赖 Kafka。

根因：docker-compose 里 Kafka 配置 KAFKA_ADVERTISED_LISTENERS: EXTERNAL://localhost:19092。 K8s Pod 通过 ExternalName Service 连到 Kafka，但 Kafka 协议特性是 broker metadata redirect — Pod 收到 broker 告诉它 "我在 localhost:19092"，Pod 去 dial 自己的 localhost，send_timeout 触发全 dropped。

修法（选项 A）：docker-compose 加 EXTERNAL_K8S listener：

kafka:
  environment:
    KAFKA_LISTENERS: PLAINTEXT://:9092,EXTERNAL://:19092,EXTERNAL_K8S://:19093
    KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092,EXTERNAL://localhost:19092,EXTERNAL_K8S://host.docker.internal:19093
  ports:
    - "19092:19092"
    - "19093:19093"

K8s Pod 用 SLINK_KAFKA_BROKERS=host.docker.internal:19093，advertised 字段会告诉 client "我在 host.docker.internal:19093"，Pod 能正确 dial 回宿主机。

反直觉教训沉淀（v0.6 反思） 通过演练 ≠ 可靠性达标。RPS / timeout / 5xx 是必要不是充分条件。 端到端对账（PG=CH drift = 0）才是充分条件。这个 silent drop 持续了 3 天才被 Day 28 端到端对账暴露。

反直觉发现（贯穿全期）

  1. "通过演练" ≠ "可靠性达标"：业务面 metric 是必要不充分。端到端对账（PG=CH）才是充分条件。v0.6 Day 27 silent drop 持续 3 天就是反例。
  2. K8s ExternalName 拦不住 Kafka 协议：metadata redirect 让 Pod client 看到 broker advertise 的地址。Kafka 在 K8s 外必须配 K8s 专用 listener。
  3. Redis INCRBY P99 1.3ms ≈ PG 1ms 同口径：跨容器网络对 KV/RDBMS 一视同仁吃 ~500μs round-trip。Redis 优势是单命令 vs PG 多段事务。
  4. 3 副本 RPS 没翻倍但 0 timeout：性能没赚到，可靠性赚到。Phase 2 解锁"滚动不漏单"而不是"RPS 多副本叠加"。
  5. distroless 强制 Go 内时序是好事：被迫把 S1-S4 全做进 Go 进程比 yaml preStop 可控 100×（可断言、可日志、可测）。

§9 工程习惯沉淀

30 天单线条迭代沉淀的 5 件套，每件都来自真实踩坑

五件套

1. 决策稿先行

每个 v0.X 启动前先写 docs/architecture/v0.X-*.md：

  • 10+ 节：背景 / 选项对比 / 决策矩阵 / 范围红线 / 灰度路径 / 风险
  • v0.4-kafka.md 13 节 / 10 决策 → 实施期间 100% 兑现，0 改动
  • 过度准备 = 实施轻松。决策时 4h 想清楚的事，实施时 4 周不用反复改

2. feature flag + git tag 双保险

  • feature flag（SLINK_EVENT_BACKEND）让灰度切流可逆：dual → shadow → kafka
  • 每次删代码前必打 git tag（v0.3-buffer-final）作回滚锚点
  • v0.4 切流删 1064 行 buffer/dualwriter 代码就是这样做的

3. 同口径 spike

选型决策必须用同口径 bench 数据，不用厂商 blog：

  • v0.4 Day 13：kgo vs sarama 同 30s pubsub bench → kgo 1.78× 决策稿封板
  • v0.5 Day 18：ch-go vs clickhouse-go vs Kafka Engine 同 rows/s + alloc/op
  • v0.6 Day 26：K8s vs nomad vs swarm 同社区 / 招聘 / 云厂商支持

4. 故障演练 + 端到端对账

v0.3 演练只看业务面，v0.4 加端到端对账。Day 28 的 Kafka silent drop 教训沉淀：

业务面 metric（RPS / timeout / 5xx）= 必要条件
端到端对账（PG count = CH count，drift 0）  = 充分条件
两者都过 → 演练真 PASS
只过业务面 → 可能是 vacuous truth（异步路径 silent drop 也察觉不到）

5. profile-first

不凭直觉调优。每次性能改动前必跑 pprof：

  • Day 6 pprof 显示 syscall 69% → 不优化 alloc，去换栈底 fasthttp
  • Day 9 调大 buffer 反向劣化 → 找瓶颈在 PG 单 flusher
  • Day 24 MV 51% CPU → DETACH 优化，6.8× 消除

3 条反 SOP（明确不做）

  • 不在一个版本动多个维度：v0.5 只动分析侧 / v0.6 只动部署侧（K8s 多副本会触发 server 无状态化连锁）
  • 不省灰度：v0.4 dual → shadow → kafka 三段式不能省。一次性切流 = 一次性回滚地狱
  • 不堆未验证候选：v0.7 候选只有真集群验 RPS 可比性 一项；sharding / mesh / HPA 全部砍掉等业务驱动
