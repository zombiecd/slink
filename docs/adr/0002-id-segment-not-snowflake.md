# ADR-0002: ID 生成使用号段模式而非 Snowflake / UUID / Hash

- **Status**: Accepted
- **Date**: 2026-05-06
- **Deciders**: @zombiecd
- **Phase**: v0.1 立项

## Context（背景）

短链系统的核心是给每个长 URL 分配一个**短**且**唯一**的标识符（短码）。短码长度直接决定：

- **用户体验**：6 位 vs 11 位短链，营销短信里差 5 个字符
- **二维码密度**：URL 越长二维码越密，扫描成功率越低
- **存储成本**：每条事件记录都包含 code 字段，长度 ×百亿事件 = TB 级差异
- **可枚举性**：短码空间越大、生成越随机，越难被爬虫遍历

我们需要决定：**短链 ID 怎么生成**。

## Options Considered（候选方案）

为统一比较，假设目标是 6-7 位 Base62 短码，对应 ID 整数空间 ≤ ~3.5 万亿。

### A. PostgreSQL `SERIAL` 直接查（朴素发号器）

- **机制**：每次创建短链查 `nextval('seq')`
- **短码长度**：6 位 OK
- **性能**：每次创建一次网络往返 → 创建 QPS 上限 ~1-2 万（PG nextval 限制）
- **可靠性**：DB 即权威源
- **多实例**：天然支持
- **致命问题**：每次创建都打 DB

### B. Twitter Snowflake

- **机制**：64 位 ID = 41 位时间戳 + 10 位机器 + 12 位序列
- **短码长度**：64 位 → Base62 编码 11 位
- **性能**：单机内存生成，4096/ms × 实例数
- **可靠性**：无中心
- **致命问题**：
  - **11 位短码超出短链可接受长度**
  - 时钟回拨会出现重复 ID
  - 节点 ID 分配本身又是问题

### C. UUID v4

- **机制**：随机 128 位
- **短码长度**：128 位 → Base62 编码 22 位
- **性能**：极快
- **致命问题**：完全不适合短链（22 位短码？）

### D. Hash(longURL) 截断

- **机制**：MD5(url) 取前 6 字节 → Base62
- **短码长度**：6 位
- **性能**：内存计算，极快
- **优点**：相同 URL 复用同短码（去重）
- **致命问题**：
  - **碰撞处理麻烦**：不同 URL 同短码 → 加随机 salt → 失去去重优势
  - 攻击者能预测短码（构造碰撞）
  - 不可枚举但**可探测**

### E. 号段模式（Leaf-segment）（最终选择）

- **机制**：服务一次取 1000 个 ID 缓存内存，用完再取
- **短码长度**：6-7 位（取决于累计 ID）
- **性能**：内存自增，单机可达千万 ID/s
- **DB 压力**：每 step_size 次创建一次 UPDATE
- **可靠性**：DB UPDATE 行锁保证全局唯一
- **多实例**：天然支持
- **致命缺点**：重启浪费当前未用完的号段（影响极小）

### F. Hash + 号段混合（v0.5+ 探索）

- **机制**：先 Hash 看是否复用，碰撞或新 URL 走号段
- **优点**：去重 + 不可预测
- **缺点**：复杂度高
- **决定**：**v0.1-v0.3 不做**，作为后续探索方向

## Decision（最终决策）

**选号段模式**（Leaf-segment 算法）。

短链 ID 生成的特殊点是**对短码长度极度敏感**。这一点直接淘汰了 Snowflake、UUID、复杂 Hash 方案。Hash 截断够短但碰撞处理代价大。号段模式：

| 维度 | 号段满足度 |
|---|---|
| 短码长度 6-7 位 | ✅ 完美控制 |
| 创建 QPS 1k+ | ✅ 内存自增不阻塞 |
| 多实例 | ✅ DB 行锁串行化取段 |
| 重启不丢号 | ✅ DB 持久化 max_id |
| 实现复杂度 | ✅ 低 |
| 可枚举攻击 | ⚠️ 顺序号 + Base62 可枚举 |

最后一行的可枚举问题，通过**短码混淆**解决（编码时 ID × 大质数 mod 全空间，使连续 ID 编出离散短码）——这是 Day 3 的 Base62 实现细节。

## Consequences（后果）

### 正面

- 6 位短码完全够用（568 亿空间），1 位都不浪费
- 创建 QPS 1k+ 不阻塞用户
- 实现极简（数百行代码 + 单测）
- DB 压力可控（每 1000 次创建一次 UPDATE）
- 后续可平滑升级为双 buffer + 异步预取（Day 3 实现）

### 负面

- 重启浪费号段（接受：号段空间几百亿，浪费 1000 个无足轻重）
- DB 是 ID 生成的中心节点（接受：v0.1 单机够用；多机房后用 biz_tag 隔离）
- 顺序短码可被枚举（用编码混淆解决，不影响算法本身）

### 风险点

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| DB 慢 → 取段慢 → 创建阻塞 | 中 | 创建 QPS 抖动 | 双 buffer + 异步预取 |
| 号段无限增长耗尽 6 位空间 | 低（568 亿）| 短码升 7 位 | 监控 max_id，到 100 亿提前升位 |
| 多机房号段冲突 | 高（如果不隔离） | ID 重复 | 每机房一个 biz_tag（如 link_dc1） |

## Revisit（何时重审）

如下条件触发重新审视：

1. 跨多机房部署成为强需求（号段需要进一步隔离设计）
2. 创建 QPS 持续 > 10w/s（DB 取段成为瓶颈，需要分片号段）
3. 短码可枚举导致真实安全事件（升级到 Hash + 号段混合）
4. 业务上需要"相同长 URL 复用短码"的去重（重新评估 Hash 方案）
