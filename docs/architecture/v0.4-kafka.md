# slink v0.4 架构 — Kafka 异步 click 写库

> 状态：📐 计划稿（2026-05-07 立）· 范围：Kafka 解耦 click_event 写入扇出 · 目标：稳态丢失率 49% → 0
> 不动：create_link 主路径 / cache 双层 / fasthttp 主端口 / Prometheus 监控 — 全部沿用 v0.3
> 配套：Day 12 journal `journal/day-12.md`（kickoff）；后续 Day 13/14/15 按本文 §10 落地

## 一、为什么 v0.4 是 Kafka

### 1.1 v0.3 真瓶颈数据

```
入口（HTTP 跳转）─────► event.Buffer (channel 50k) ─────► PG 单 flusher (COPY FROM)
   ~93k/s 入速              缓冲层                          ~62k/s 出速
                                                              ↓
                                                          稳态溢出 49%
```

Day 9 已经验证：**调参解决不了**。把 buffer 从 50k/2k/500ms 调到 100k/5k/200ms，RPS -17.6% / P99 +130% — 因为 PG 单 flusher batch=5000 一次 COPY 耗时拉长 → 单 goroutine 阻塞 → 出口反而变慢。

**根因结构性：** PG 是 OLTP 数据库，扛"每秒 9w 事件流"不是它擅长的事。click_event 表 + 月分区 + COPY FROM 已经把 PG 单点能榨到极限了，再优化空间小。

### 1.2 v0.4 目标

| 维度 | v0.3 实测 | v0.4 目标 |
|---|---:|---:|
| 入速（HTTP 跳转）| 93k/s | ≥ 93k/s（不退步）|
| 缓冲层稳态丢失 | **49%** | **< 1%** |
| 端到端延迟（click 落库可见）| ~500ms | < 5s（异步可接受）|
| 主路径 P99 | 24.7ms | 不退步 |
| 故障域 | 1（PG 挂全挂）| 2（Kafka 挂主路径不退；PG 挂只影响 consumer）|

**关键不变量**：主路径（`POST /api/links` + `GET /:code` + 302）**绝对不能为了 click 写入而退步**。click 是观察用，不是业务关键路径。

## 二、为什么不是 X

### 2.1 不是"多 PG flusher"

考虑过：把 1 个 flusher 拆 4 个，并行 COPY。

**否决理由：**
- PG 写入扇出本质是连接池竞争 + WAL 串行。多 flusher 顶多吃满 CPU，不能突破 PG 单实例写入 IOPS。
- 测过类似场景：4 flusher 提升 ~40%（不是 4x），瓶颈平移到 PG 自身 fsync。
- 架构上没解耦 — PG 抖动还是直接打到主路径。

### 2.2 不是"ClickHouse 直写"

考虑过：跳过 PG，click_event 直接写 ClickHouse（OLAP 分析友好）。

**否决理由：**
- ClickHouse 是 OLAP，不擅长高频小批量写（推荐每 100ms+ 批量）。
- 没有"削峰"层，主路径还是受 ClickHouse 写抖动直接影响。
- 但 v0.5+ 会做：**Kafka → ClickHouse consumer 批写**，是 v0.4 的天然下一步。

### 2.3 不是"加 PG 分库"

考虑过：click_event 走单独的 PG 实例，不和主表挤同一池。

**否决理由：**
- 不解决"PG 单 flusher 出速 ~62k/s"的根本问题，只是让影响别人的范围小一点。
- 运维复杂度上升（多一台 PG），收益不对等。

### 2.4 为什么是 Kafka

- **削峰天然**：producer 写入只看 Kafka broker，不看 PG。Kafka 单 broker 100k+/s 是入门级。
- **故障域分离**：Kafka 挂了 → 主路径降级为"丢 click 但不影响跳转"；PG 挂了 → consumer 暂停，事件在 Kafka 排队，PG 恢复后追上。
- **下游可扩展**：v0.5 加 HLL UV / TopK 实时聚合时，多挂一个 consumer 就行，不需要碰 producer。
- **业界标准**：写"短链事件流"几乎所有公司都是 Kafka，简历叙事正好对线。

## 三、模块图（v0.3 → v0.4）

### v0.3 现状

```
            ┌─────────────────────────┐
GET /:code ─►│ redirect handler         │
            │   ↓ enqueue ClickEvent   │
            │ event.Buffer (50k chan)  │
            └────────────┬────────────┘
                         │
                         ▼
                  ┌──────────────┐
                  │ flusher GR   │  batch COPY FROM
                  │ (单 goroutine)│
                  └──────┬───────┘
                         ▼
                  ┌──────────────┐
                  │ PG click_events│
                  └──────────────┘
```

### v0.4 目标

```
                    ┌─────────────────────────┐
GET /:code ────────►│ redirect handler        │
                    │   ↓ producer.Send       │
                    │ event.KafkaProducer     │  ← 替换 event.Buffer
                    └────────────┬────────────┘
                                 │ async send (sarama / kgo)
                                 ▼
                          ┌─────────────┐
                          │   Kafka     │  topic: slink.click_events
                          │  (1 broker  │  partition: 4 (key=code)
                          │   单机起步) │  retention: 7d
                          └─────┬───────┘
                                │
              ┌─────────────────┴──────────────────┐
              │                                    │
              ▼                                    ▼
     ┌──────────────────┐                ┌────────────────────┐
     │ cmd/consumer/    │                │ (v0.5 占位)         │
     │ ClickEventConsumer│                │ ClickHouse / HLL    │
     │   ↓ batch COPY   │                │   sink consumer     │
     │ PG click_events  │                └────────────────────┘
     └──────────────────┘
```

**关键变化：**
1. `event.Buffer` 整体替换为 `event.KafkaProducer`（接口 `Eventer.Enqueue` 不变 → handler 不动）
2. 新增 `cmd/consumer/main.go` — 独立进程 / 独立 binary
3. 新增 docker-compose 服务：`kafka` + `kafka-ui`（kowl/redpanda console）

## 四、Topic / Partition / Key 设计

### 4.1 Topic

| 名 | 用途 | retention |
|---|---|---|
| `slink.click_events` | redirect 跳转事件流 | 7 天 |

只有 1 个 topic，v0.4 范围内不拆。create_link 不进 Kafka（写量低 + 强一致性需求）。

### 4.2 Partition

**4 个 partition** 起步。理由：
- 单 broker 单机版本，partition 数 = 并行消费上限。
- 4 个让未来加 1-2 个 consumer 时还有余量。
- 太多（如 64）让单机 broker 元数据开销不必要。
- 加 partition 可以在线，减不能 → 起步保守。

### 4.3 Key 选择 = `code`（短链码）

**为什么用 code 不是 event_id：**
- **同一 code 落同 partition** → consumer 端可以做"按 code 聚合"的状态计算（如 v0.5 的 HLL UV）不跨 partition 通讯。
- event_id（UUID）作为 key 会让事件均匀打散，丢失"按 code 聚合"的局部性。
- 缺点：热门 code 可能造成 partition 热点。监控加一条 partition lag 告警，热点出现再迁。

**单调性顾虑：** code 是 base62(id × P mod N)，看起来无序，但相邻请求的 ID 是单调的 → 经过位置混淆后 hash 仍然均匀，partition 分布健康。

### 4.4 消费语义

**at-least-once**（默认 Kafka 语义）。click_event 表已有 `event_id` UUID 主键，consumer 重复投递走 `INSERT ... ON CONFLICT (event_id, ts) DO NOTHING` 自动去重。

不上 EOS（exactly-once）：
- click_event 容忍重复（看分析趋势不差几条）
- EOS 配置复杂，吃性能

## 五、Producer 侧设计

### 5.1 库选型

候选：`IBM/sarama` vs `twmb/franz-go` (kgo) vs `confluent-kafka-go`。

**已选 kgo (franz-go)**（Day 13 spike 决策，2026-05-07）。详见 `docs/concepts/kafka-client-choice.md`。

实测对比（30s 单 producer goroutine / 100B JSON / lz4 / acks=leader / linger=5ms）：

| | sarama 1.45.2 | **kgo 1.19.5** | 差距 |
|---|---:|---:|---:|
| RPS | 443,842 | **788,183** | kgo 1.78× |
| ack p99 | 30.4 ms | 31.1 ms | 持平（broker bound）|
| heap inuse Δ | +8.6 MB | +19.0 MB | sarama 省 10MB |
| mallocs 30s | 134 M | 263 M | kgo 2× |

理由排序：
1. 吞吐 1.78×（client 上限更高）
2. API context-first + 函数 option（与 fasthttp router 风格一致）
3. 纯 Go 无 cgo（与 confluent-kafka-go 区分）
4. p99/max 与 sarama 持平（broker bound）

风险：mallocs 2× 但 heap 稳定，Day 14 全链路压测验证 alloc/req 退步是否 > 10%。回退预案：替换 `internal/event/kafka.go` 一个文件，接口 `Eventer` 不变。

### 5.2 接口对齐

`event.Eventer` 接口保持不变：

```go
type Eventer interface {
    Enqueue(ctx context.Context, evt ClickEvent) error
}
```

`KafkaProducer` 实现这个接口 → handler 不知道底层是 channel 还是 Kafka，**replace-only 改动**。

### 5.3 Producer 关键参数

| 参数 | 起步值 | 理由 |
|---|---|---|
| RequiredAcks | `LeaderAck`（不是 AllISR） | 单 broker 单机，没有 ISR 概念，LeaderAck 足够 |
| Compression | `lz4` | 短链事件 ~100 字节，压缩比一般，lz4 CPU 开销最低 |
| Async batching window | 10ms | 削峰主要靠 Kafka 自己，producer 端 batch 短一点保延迟 |
| Max in-flight | 5 | sarama 默认 5，平衡吞吐和顺序保证 |
| Linger | 5ms | 短 linger 让小流量也能批 |

### 5.4 失败/降级

**Kafka 不可达时主路径绝对不能阻塞。** 三道闸：

1. **producer 同步发送 ctx.Deadline = 100ms** → 100ms 内没回执则 `Enqueue` 返回 error
2. **handler 拿到 error 后只 `slog.Warn`，不阻塞跳转** → 跳转成功率 = 99.99%（同 v0.3）
3. **producer 内部 buffer 满（队列溢出）** → drop + counter `slink_kafka_producer_dropped_total++`，监控告警

**故障演练**（v0.4 完工前必跑）：
- 跳转压测中途 `docker stop kafka` → 监控应该看到 dropped 飙升，但 RPS 不掉
- 重启 Kafka → producer 自动重连，dropped 归零

## 六、Consumer 侧设计

### 6.1 进程结构

```
cmd/server/main.go          ← 主 HTTP server（不变）
cmd/consumer/main.go        ← 新增：click_event consumer
internal/event/buffer.go    ← v0.4 删（被 KafkaProducer 替代）
internal/event/kafka.go     ← 新增：KafkaProducer
internal/event/consumer.go  ← 新增：ClickEventConsumer + batch COPY
```

两个独立 binary。docker-compose 起 server + consumer 两个 service，**故障域天然分离**。

### 6.2 Consumer 处理流程

```
Kafka topic ──► consumer poll batch (max 1000 条 / 100ms)
                  │
                  ├─ 解码（JSON / protobuf？v0.4 起 JSON，v0.5 看性能要不要切 proto）
                  ├─ 累积 batch
                  ├─ COPY FROM 写 PG（同 v0.3 flusher）
                  ├─ commit offset
                  └─ 失败：不 commit offset，下一轮 poll 重读
```

### 6.3 Consumer 配置

- **consumer group**：`slink.click_events.pg_writer`（命名前缀让 v0.5 加 HLL consumer 时不冲突）
- **partition assignment**：`range`（默认）或 `round-robin`（多 consumer 时考虑），起步默认
- **auto-commit = false**，手动 commit 在 batch 写 PG 成功之后
- **session timeout**：30s（默认）

### 6.4 监控

新增 metrics：
- `slink_kafka_producer_sent_total` / `slink_kafka_producer_dropped_total` / `slink_kafka_producer_errors_total`
- `slink_kafka_consumer_lag_seconds`（partition lag 监控）
- `slink_kafka_consumer_processed_total` / `slink_kafka_consumer_errors_total`

Grafana dashboard 加一行 "Kafka & Consumer" panel。

## 七、故障模式 + 降级

| 故障 | v0.3 行为 | v0.4 行为 | 改善 |
|---|---|---|---|
| **PG 抖动** | 主路径不卡（cache hit），但 click 全部 dropped | 主路径不卡，click 在 Kafka 排队，PG 恢复后追上 | ✅ 显著 |
| **PG 挂超过 retention 7d** | n/a | Kafka 自然过期丢失 7d 之前的 click | 可接受（click 是观察用）|
| **Kafka broker 挂** | n/a | producer 100ms timeout 后 drop + warn，主路径不退步 | 同 v0.3 dropped 行为，但故障域分离 |
| **Consumer 挂** | n/a | Kafka 排队，重启 consumer 自动追上 | 新故障域，可独立修复 |
| **网络分区** | 主路径不影响（lo 通讯）| 同 v0.3，因为 producer 失败也是 drop+warn | 同 |

**关键判断**：v0.4 的"故障表面积"看起来比 v0.3 大（多了 Kafka + Consumer 两个组件），但**主路径的故障表面积没变** — 主路径只 ↑ 一个 Kafka producer 调用，失败路径已经是 drop+warn。所以"主路径稳定性"不退步。

## 八、灰度迁移方案

不能一刀切，分 3 步：

### 8.1 Step 1：双写期（Day 14 落地）

`event.Buffer`（v0.3 老路径）和 `event.KafkaProducer`（v0.4 新路径）**同时跑**。

```go
type Eventer interface { Enqueue(ctx, evt) error }

type DualWriter struct {
    primary   Eventer  // KafkaProducer
    secondary Eventer  // event.Buffer（v0.3 老）
}

// Enqueue 同时投到两边，primary 失败不影响 secondary
```

期间：
- 监控 producer 成功率 / consumer lag
- 对账：consumer 写入 PG 的事件数 vs Buffer 直写的事件数
- 如果 producer 成功率 < 99.9% → 切回单写 Buffer

### 8.2 Step 2：影子期（Day 15 早期）

producer 走 Kafka + consumer 写 **影子表** `click_events_shadow`，不动 v0.3 主表 `click_events`。

期间：
- 对比影子表 vs 主表的事件数 / 字段一致性
- 跑 7 天观察 lag / 故障演练 / Kafka 故障重启等场景

### 8.3 Step 3：切流（Day 15 晚 / Day 16）

- consumer 改写主表 `click_events`
- 老的 Buffer 路径关闭
- 删 `event.Buffer` 代码
- 删影子表

**回滚预案**：feature flag `SLINK_EVENT_BACKEND=buffer|kafka|dual`。生产事故可在 1 分钟内切回 buffer。

## 十、范围红线（v0.4 不做）

明确不做的事，避免 scope creep：

- ❌ **EOS（exactly-once）** — at-least-once + DB unique 去重已经够用
- ❌ **多 broker 集群** — 单 broker 单机起步，多 broker 留 v0.5 / 真上线再说
- ❌ **跨数据中心 replication** — 同上
- ❌ **Kafka Streams / KSQL 实时聚合** — HLL UV / TopK 留 v0.5
- ❌ **schema registry** — JSON 消息体起步，v0.5 看是否切 proto 再加
- ❌ **替换 create_link 路径** — 创建短链是低频强一致写，不进 Kafka
- ❌ **替换主路径业务接口** — handler / cache / fasthttp / SSRF / 幂等 全部不动

任何"v0.4 顺手做一下..."的诱惑都先入待办表，不做。

## 十一、决策记录（Day 12）

按本架构稿落地的关键单方决策（用户已对齐"主线 = Kafka，今晚仅写 plan 不动 code"）：

| 决策 | 选项 | 选了 | 理由 |
|---|---|---|---|
| 主线方向 | Kafka / Kratos / K8s+OTel | **Kafka** | v0.3 真瓶颈是 PG 单 flusher，路线图主战场 |
| 库选型 | sarama / kgo / confluent | **kgo 待实测** | API 现代 + 纯 Go；Day 13 spike 决定 |
| Topic 数 | 1 / 多 | **1（slink.click_events）** | v0.4 只解 click 一个事件流，不扩面 |
| Partition 数 | 1 / 4 / 16 | **4** | 平衡并行度 + 元数据开销 |
| Key 选择 | event_id / code | **code** | 同 code 聚合不跨 partition |
| 消费语义 | EOS / at-least-once | **at-least-once + DB 去重** | 复杂度 vs 性价比 |
| 故障降级 | 阻塞 / drop+warn | **drop+warn** | 主路径不能因 click 退步 |
| 迁移方式 | 一刀切 / 双写期 / 影子期 | **双写 → 影子 → 切流 三步** | 生产级谨慎度 |
