## 一、为什么 v0.6 是 K8s（多副本 + server 无状态化）

### 1.1 v0.5 留的真问题

v0.5 收口（Day 25）后链路结构：

```
       client
         │
         ▼
  fasthttp（单进程，单机）
         │
   ┌─────┼─────┐
   ▼     ▼     ▼
  PG    Redis  Kafka ──► PG Consumer + ClickHouse Engine
                          （已多消费者组，已故障域分离）
```

后端侧（PG / Redis / Kafka / CH）已经在 Day 16 切流 + Day 23 drill 验证过故障域 3 分离（v0.4 D5 + v0.5 drill A/B/C），但 **server 本身仍是单进程单机**。

具体留的问题：

1. **单点故障**：server 进程挂 = 100% 流量挂；机器重启 = 服务 down。后端再多 9 也救不了 server。
2. **无法滚动升级**：现在升 server 必须停服。生产环境每次发布 = 一段全量漏。
3. **故障演练有盲区**：v0.4 D5 / v0.5 drill 都是杀**后端**（PG / CH），没杀 server 自己。server 自己被 SIGTERM 时，in-flight 请求 / Kafka producer buffer / L1 dirty 状态怎么处理，现在没答案。
4. **横向扩容只能换机**：当前提升单机性能（fasthttp + L1 + 进程内号段），但单机硬上限就在那，无法靠加机器扩。

### 1.2 v0.6 目标

| 维度 | v0.5 | v0.6 目标 |
|---|---|---|
| 后端故障域分离 | ✅ 3 个（PG / CH / Kafka）| ✅ 不退步 |
| 主路径 RPS（多副本聚合）| 93k 单机 | **≥ 93k**（不退步；多副本聚合后大概率涨）|
| 主路径 P99 | 24.7ms 单机 | **≤ 30ms**（K8s 网络层 + service mesh 可能加 1-2ms）|
| **新增：server 多副本** | n/a（单进程）| **replicas=3 + 滚动 0 漏** |
| **新增：滚动升级** | 不能 | **kubectl rollout restart 0 漏 0 timeout** |
| **新增：server 无状态化** | id 号段进程内 / L1 进程内 / 停机不优雅 | **id 号段中心化 / L1 接受短期 miss / preStop hook + drain** |
| 顺手做：OTel trace | n/a | trace 链 client→server→{PG,Redis,Kafka,CH} |
| 顺手做：/api/stats/* 鉴权 | 无鉴权（admin endpoint 暴露）| K8s NetworkPolicy + Ingress AuthN |
| 顺手做：recon-fixture.sh 重写 | v0.5 R1 时间窗 bug 留账 | 重写 |

**关键不变量**：v0.5 已建好的"分析侧不影响主路径"原则继续。**K8s 多副本只动 server 层**，PG / Redis / Kafka / CH 仍在 host 网络（Phase 1）；K8s 化它们留 v0.7+。

## 二、为什么不是 X

### 2.1 不是"docker swarm / nomad"

考虑过：swarm 是 docker 自带，学习成本最低；nomad 是 hashicorp 系，单 binary。

**否决理由：**
- swarm 已实质半弃维（社区活跃度逐年降）
- nomad 生态远小于 K8s（社区 / 招聘资源 ≈ 50:1）
- K8s 是事实标准，所有云厂商托管（EKS / GKE / ACK / TKE）；学完一次终身受益
- minikube / kind / k3d 本地起栈 5 分钟，门槛已经不高

### 2.2 不是"暂时不做无状态化，只起多副本"

考虑过：先把 server 跑 K8s 上 replicas=3，无状态化连锁问题（id 号段共享 / L1 一致性）留 v0.6.1。

**否决理由：**
- id 号段不共享 = 多副本会发 **重复 ID** 给客户端 → 数据库 PK 冲突 / 客户端拿到错误短链。这是**正确性 bug**，不是性能问题，不能延期。
- L1 不一致 = Pod A 写后 Pod B 读到旧值，业务可承受短期 miss（短链查询）但**不能完全不解决**（必须有"接受短期 miss"的明确决策 + TTL 上限）。
- Pod 滚动不优雅 = 每次发布漏一段，验证不过"0 timeout"硬指标。

三个连锁问题都是**多副本的前置必答题**，不是可选项。

### 2.3 不是"先做 OTel，K8s 留更后"

考虑过：v0.5 retro §4 也提到 OTel 是"v0.6 顺手做"，那能不能反过来？

**否决理由：**
- OTel 是基建增量（在现有 server 上加 SDK + collector）—— 单机就能做，不依赖 K8s
- K8s 是部署侧大改 + 三连锁正确性问题—— 一次只动一个维度
- 实际上做 K8s 时，OTel 走 sidecar 注入更顺；先 OTel 再 K8s = 第二阶段还要重做注入
- v0.5 retro §4 拍板"v0.6 主战场 = K8s，OTel 顺手"—— 已对齐决策，不单方面修改（meta-cognition §1）

### 2.4 不是"先做 ClickHouse sharding / replication"

CH 是 v0.5 单机起步，sharding/replication 看似"补完 v0.5"。

**否决理由：**
- v0.5 retro §6 已答：单机 CH 5M 行 UV 6.6ms，没瓶颈。横向扩是 1B 行规模才有意义。
- sharding 必须有 K8s（StatefulSet + headless service + sharding key）才有运维基础。没 K8s 强行 sharding = 手糊运维灾难。
- v0.6 K8s + 无状态化结束后，v0.7 起做 PG/Redis/CH 的 K8s 化 + 横向扩，顺序更顺。

### 2.5 不是"做 service mesh（istio / linkerd）"

K8s 自带 Service / Ingress 已经能解决 v0.6 范围内的所有问题（多副本 LB / 滚动 / NetworkPolicy）。istio / linkerd 是更深的可观测性 + 流量管理，v0.6 不需要。

留 v0.7+。

### 2.6 为什么是 K8s（这个时间点 / 这个项目）

- **v0.5 retro §4 已拍板**：v0.6 主战场 K8s 多副本 + 无状态化（meta-cognition §1 不单方面修改对齐决策）
- **工程演进完整**：单机 86k → K8s 多副本 + 滚动 0 漏 = 完整迭代链路
- **三大连锁问题是真高质量学习点**：id 号段中心化、cache 一致性、优雅停机—— 这是分布式系统教材的核心三件套，做过有体感
- **后端已就位**：v0.5 故障域 3 分离已验证，K8s 只动 server 层，blast radius 可控

**关键设计**：

1. **K8s 范围只覆盖 server**。PG / Redis / Kafka / CH / Consumer 全部留 host 网络（host.docker.internal 或 ExternalName Service）。这是 v0.6 范围红线之一（§5）。
2. **每个 Pod 有自己的 L1 cache + id 号段缓冲**。Pod 间不共享。L1 接受短期 miss（详 §8.2）；id 号段从 PG 中心化拿一段（详 §8.1）。
3. **3 副本是起点**，不是上限。滚动验证通过后，replicas 可调任意值。
4. **OTel collector** Phase 4 加，trace 链路覆盖 client→Ingress→Pod→{PG, Redis, Kafka, CH}。
5. **/api/stats/* 鉴权** Phase 4 加，K8s NetworkPolicy 限只 admin-only Pod 可访问 + Ingress AuthN（basic auth 或 oauth2-proxy）。

## 五、范围红线（v0.6 不做）

明确不做的事，避免 scope creep：

- ❌ **K8s 化 PG / Redis / Kafka / ClickHouse** — 留 host 网络，Pod 直连。K8s 化它们留 v0.7+（StatefulSet + PVC + sharding 是另一个赛道）
- ❌ **CH sharding / replication** — 单机够用（v0.5 retro §6 已答）
- ❌ **service mesh（istio / linkerd）** — Service + Ingress 够用
- ❌ **多 K8s 集群 / 多 region** — 单 cluster 单 namespace
- ❌ **HPA / VPA 自动扩缩** — replicas 写死先验证手动滚动；自动扩缩是 v0.7+
- ❌ **改 v0.4 KafkaProducer / v0.5 ClickHouse 链路** — 任何改动需要回 v0.4/v0.5 架构稿对齐
- ❌ **改主路径** — fasthttp / cache 路径不动；只在 Pod 启停时机加 hook
- ❌ **持久化 L1 cache** — L1 接受 Pod 重启清零（详 §8.2 决策）
- ❌ **新业务功能** — v0.6 范围只到"部署侧"

任何"v0.6 顺手做一下..."的诱惑（OTel / 鉴权 / recon-fixture 重写除外，已纳入计划）都先入待办表，不做。

## 八、三大未知风险方案对比（核心）

v0.6 的难度全在这三条。任何一条决策错 = Phase 2 重做。

### 8.1 id 号段共享

**问题**：v0.3 双 buffer 是单进程内的（基于 `internal/id`），多 Pod 会发重复 ID。

**候选方案**：

| 候选 | 机制 | 优点 | 缺点 | 复杂度 |
|---|---|---|---|---|
| **A：PG 号段表** | 新表 `id_batches`，每个 Pod `BEGIN; SELECT next, next+10000 FOR UPDATE; UPDATE; COMMIT;` 拿一段 | 强一致 / 已有 PG 依赖 / 可审计 | PG 写压力 +10000/N（N=batch 大小）| 低 |
| **B：Redis INCRBY** | `INCRBY id_seq 10000` 拿 [old+1, old+10000] 段 | 性能好（μs 级）/ 无锁 | Redis 重启丢号段（可接受 / 但需 AOF）/ 已有 Redis 依赖 | 极低 |
| **C：Snowflake / Sonyflake** | Pod 启动时申请 worker_id（PG 表 / Redis SET NX）+ 本地时间序列 | 完全去中心 / 高吞吐 | worker_id 分配本身仍中心化 / 时钟回拨陷阱 / 复杂度收益不匹配 | 中 |

**评估矩阵**：

| 维度 | A PG | B Redis | C Snowflake |
|---|---|---|---|
| 一致性 | 强 | 强（AOF）/ 弱（默认）| 强 |
| RPS 影响（拿号段时）| 1ms tx | 50μs INCR | μs 级（本地）|
| 拿号段频率 | batch=10000 → ~10 RPS（93k 时）| 同 | n/a |
| Pod cold-start | 1 PG round-trip | 1 Redis round-trip | 1 Redis SET NX + 启动后本地 |
| 实现 LOC | ~80 | ~50 | ~150 |
| 故障域 | PG 挂 = 拿号段挂 | Redis 挂 = 拿号段挂 | worker_id 服务挂 = cold-start 挂（已运行 Pod 不影响）|

**Day 26 拍板：B Redis INCRBY**（带 spike 兜底）

理由：
1. **性能**：50μs vs PG 1ms，号段耗尽时不卡 P99
2. **依赖最少**：v0.3 已有 Redis 在，不引入新组件
3. **代码量最小**：50 行 vs PG 80 / Snowflake 150
4. **故障域已被 v0.4 验证**：Redis 挂时主路径如何降级，v0.3 已有方案
5. **AOF 持久化**：production Redis 默认 AOF every-second，重启丢秒级号段段可接受（号段表设计上号段是单调递增，丢一段相当于跳号，不重复）

**Spike 兜底**：Day 27 spike 单独验 Redis INCRBY 在 3 副本同时拿号段时的 P99（预期 < 100μs）。如果不达预期，回退 A PG 号段表。

#### Day 27 spike 实测校准

**实测**（3 workers × 200 acquires，docker compose host network，`internal/store/segment_redis_test.go::TestRedisSegmentSource_ConcurrentSpike`）：

| 指标 | 实测 | 原 §8.1 估值 |
|---|---|---|
| P50 | 474μs | 50μs |
| P99 | 1295μs | < 100μs |
| Max | 3451μs | — |
| 0 重复 / 0 漏 | ✅ | — |

**估值偏差根因**：原 §8.1 引用「Redis INCRBY 50μs」是裸进程内 Redis 数字。docker compose host network TCP round-trip 主导（~470μs），Redis 服务端单 INCRBY 处理只 ~10μs（占总耗时 < 5%）。**网络不是 Redis 本身的锅**。

**业务可接受性分析**（决策保留 Redis，**不回退 PG**）：
1. **号段批量掩盖**：stepSize=1000 + 双 buffer 异步预取，93k RPS 时拿号段频率 ~93/s。拿号段阻塞 P99 1.3ms 被异步路径完全掩盖，**不上主路径 P99**
2. **PG 同样吃 docker network**：原决策稿对比的 PG 1ms 是同样跨 docker 网络的数字，但 PG 还要 `BEGIN/UPDATE...RETURNING/COMMIT` 三段，实际 P99 大概率 > 2ms。Redis 1.3ms 仍占优
3. **Phase 3 多副本回归会再验**：拿到端到端 RPS 时如果发现主路径 P99 受影响，再 pivot 也来得及

**修订 §8.1 spike 标准**：从「P99 < 100μs」改为「**P99 < 2ms**」（基于实测拓扑，仍能覆盖 90% 异常场景）。

**Day 27 拍板（校准后保留）：B Redis INCRBY**

**留口**：Snowflake 是 v0.7+ 真正全去中心化时再做（现在不值）。

### 8.2 L1 cache 一致性

**问题**：v0.3 进程内 L1 命中 99.98% 是单进程独享。Pod A 写后 Pod B 读到旧值会发生：
- 创建短链：`POST /api/links` → Pod A 写 PG + 写 Pod A L1 → Pod B 第一次 GET 这个 code → Pod B L1 miss → 走 Redis → Redis miss → 走 PG → 命中 → 回填 Pod B L1
- 删除短链（v0.6 范围内不做删除接口，但理论存在）：Pod A 删 → Pod B L1 仍有旧值 → 短期内 Pod B 可读到已删除短链

**候选方案**：

| 候选 | 机制 | 优点 | 缺点 |
|---|---|---|---|
| **A：接受短期 miss + L1 TTL 上限** | 创建后 Pod B 第一次读 = L1 miss + 走 Redis（已写入）→ 命中。L1 自然 TTL 5s 上限保证最终一致。| 0 改动 / 性能不退步 / 业务可接受（短链查询不要求强一致）| 删除场景有 5s 不一致窗口（v0.6 范围内不做删除）|
| **B：Redis pub/sub 失效广播** | Pod A 写 PG 后 publish 到 channel `cache_invalidate`，所有 Pod sub 后删本地 L1 | 一致性接近实时（< 100ms）| +1 Redis 链路 / Pod sub goroutine / 网络分区时 sub 漏消息 |
| **C：分布式 cache（去 L1，全走 Redis）** | 删 L1，所有 cache 走 Redis | 强一致 | 99.98% L1 命中带来的 P99 优势全丢 / 退到 v0.2 性能 |

**评估矩阵**：

| 维度 | A 接受 miss | B pub/sub | C 去 L1 |
|---|---|---|---|
| 一致性窗口 | < 5s（TTL 自然过）| < 100ms | 0 |
| 性能影响 | 0 | sub 链路 +微小 | -50%+ RPS（L1 命中 99.98% 全丢）|
| 实现 LOC | 0（已有 TTL）| ~120 | -200（删 L1 代码）但全栈测试要重做 |
| 业务可接受 | ✅ 短链查询不要求强一致 | ✅ | ✅ |
| 复杂度 | 极低 | 中 | 高（牵扯主路径）|

**Day 26 拍板：A 接受短期 miss + L1 TTL 上限**

理由：
1. **业务语义匹配**：短链是"创建 → 多次读"的写少读多场景，新创建的短链第一次读延迟到 PG 拿一次（毫秒级），用户感知不到
2. **0 改动主路径**：v0.3 L1 设计已经有 TTL，不需要新代码
3. **性能不退步**：99.98% L1 命中保留
4. **v0.6 范围红线 §5**：不动主路径
5. **退化路径清晰**：如果未来真有强一致需求（比如 link expiration 业务），再走 B pub/sub，不影响 v0.6

**Spike 兜底**：Day 27 spike 多 Pod 创建短链后立即跨 Pod 读，验证 P99 仍 < 30ms（Redis miss + PG 命中 + 回填）。如果 Redis miss 率异常高，再考虑 B。

**留口**：删除接口不在 v0.6 范围（红线 §5）。如果未来加，必须配 pub/sub（B）。

### 8.3 Pod 滚动 0 漏 / 优雅停机

**问题**：K8s 滚动升级 = 老 Pod SIGTERM + grace period + SIGKILL。期间：
- Service endpoints 还在路由流量到老 Pod（K8s 摘流是异步的，有 1-2s 滞后）
- 老 Pod 的 fasthttp 还在处理 in-flight request
- 老 Pod 的 KafkaProducer 还有未 flush 的 batch
- L1 cache 状态会丢（接受，§8.2）
- id 号段缓冲未用完（接受，号段是单调递增的）

如果不优雅停机：
- in-flight 请求被强杀 → 客户端 5xx
- KafkaProducer batch 未 flush → click event 漏（v0.4 切流时辛苦做的 0 漏可能毁）

**候选方案**（不是单选，是**组合拳**）：

| 步骤 | 机制 | 必做 / 可选 |
|---|---|---|
| **S1：preStop hook** | K8s 在 SIGTERM 前调用 `sleep 5` 给 service controller 摘流时间 | 必做 |
| **S2：readinessProbe 主动 fail** | 进程收到 SIGTERM 后立即让 `/healthz` 返 503，加速 K8s 摘流 | 必做 |
| **S3：fasthttp graceful shutdown** | `server.Shutdown()` 等 in-flight request 完成（带超时）| 必做 |
| **S4：KafkaProducer flush** | `producer.Close()` 触发同步 flush 等所有 batch ack | 必做 |
| **S5：drain interval** | 配置 `terminationGracePeriodSeconds=30s` 给 S1-S4 留时间 | 必做 |
| **S6：Pod 数量保证** | `maxSurge=1 maxUnavailable=0` 滚动策略 = 总有 N+1 Pod 过渡期 | 必做 |
| **S7：rolling 速率限制** | rolling 间 1 Pod 不要并发 | 必做 |

**优雅停机时序图**：

```
t=0      : K8s 决定杀 Pod，发 preStop（sleep 5）+ 标记 endpoint NotReady
t=0~5s   : Ingress / Service 把流量切到其他 Pod（最长 1-2s 完成）
t=5s     : preStop 完成 → SIGTERM 发给进程
t=5s     : 进程收 SIGTERM → readinessProbe = 503 / fasthttp.Shutdown() / KafkaProducer.Close()
t=5~25s  : drain in-flight（fasthttp）+ flush Kafka batch（producer）
t=25s    : 所有完成 → 进程退出 0
t=30s    : terminationGracePeriodSeconds 到 → 如未退出 SIGKILL（兜底）
```

**Day 26 拍板：S1-S7 全做**

理由：
1. v0.6 硬指标 = 滚动 0 漏 0 timeout，不能省任何一步
2. 每步都是 K8s 标准实践
3. 实现 LOC 总和 ~150 行（fasthttp.Shutdown / producer.Close 大头），可控

**Spike 兜底**：Day 27 spike 跑 1 Pod kill（先不滚动，测单点 SIGTERM 行为）+ Day 30 滚动验证。

**留口**：超大流量（10 万 RPS+）下 fasthttp drain 时间可能超 25s，那时候要把 grace period 调大或者降低单 Pod RPS（加副本）。v0.6 范围内 93k RPS / 3 副本 = 31k/Pod，drain 25s 应该够。
