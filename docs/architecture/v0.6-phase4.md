## 一、为什么 Phase 4 不另起 v0.7

### 1.1 v0.6 Phase 0-3 完成后的留口

v0.6 收口（Day 28）后留下 4 类部署侧未补全：

```
                            [client]
                                │
                                ▼
                     ┌──────────────────┐
                     │   ❌ 无 Ingress   │ ← (a) /api/stats/* 无鉴权
                     └────────┬─────────┘
                              │
                  ┌───────────┼───────────┐
                  ▼           ▼           ▼
           [Pod 1]      [Pod 2]      [Pod 3]
           ❌ 无 OTel     ❌ 无 OTel    ❌ 无 OTel  ← (b) trace 链路缺失
                  │           │           │
                  └─────┬─────┴─────┬─────┘
                        │           │
                  PG/Redis/Kafka  CH    ← (c) 无 NetworkPolicy（任何 Pod 可访问任何后端）
                        │
                        ▼
                  cmd/consumer (host)   ← (d) 单点 host 进程未 K8s 化
```

### 1.2 为什么 Phase 4 顺手做不另起 v0.7

**v0.6 retro §4 已拍板**：v0.7 留给「真集群（GKE/EKS）验 RPS 可比性」+「ClickHouse sharding/replication」+「HPA/VPA」等更大改动。

**Phase 4 4 项都是 K8s 部署侧补全**——与 v0.6 主体一致，分到 v0.7 会让"部署侧"故事断线。

| 维度 | v0.6 Phase 0-3 | Phase 4 |
|---|---|---|
| 主线方向 | server K8s + 三连锁 | K8s 部署侧补全 |
| 改动范围 | server 进程 + Deployment yaml | OTel SDK + sidecar + NetworkPolicy yaml + Ingress + 1 个 consumer Deployment |
| 风险 | 三连锁正确性（id 重复）| 配置层（无主路径正确性风险）|
| 简历叙事 | 多副本 + 滚动 0 漏 | 可观测性 + 部署侧安全 + 全 K8s 化 |

**单一维度原则**（meta-cognition §1）：v0.7 主战场已锚定真集群验证，Phase 4 提前做完 K8s 收口，让 v0.7 焦点纯净。

### 1.3 Phase 4 目标硬指标

| 维度 | v0.6 Phase 3 | Phase 4 目标 |
|---|---|---|
| OTel trace 覆盖 | ❌ | ✅ client → Ingress → Pod → {PG, Redis, Kafka, CH} 全链路 |
| /api/stats/* 鉴权 | ❌（裸暴露） | ✅ Ingress AuthN（basic auth 或 oauth2-proxy）|
| Pod 间网络隔离 | ❌（任意 Pod 可访问任意后端） | ✅ NetworkPolicy 白名单（slink-server only 访问 PG/Redis/Kafka/CH）|
| consumer 部署形式 | host 单点 | ✅ K8s Deployment（replicas=2 起步 + Kafka group rebalance）|
| 滚动 0 漏不退步 | 2.22M reqs / 0 timeout | ✅ 不退步（演练复跑同档）|
| 端到端 PG⇄CH drift | 3× 0.0000 | ✅ 不退步（含 consumer K8s 化场景）|
| recon-fixture.sh 时间窗 bug | 留账 | ✅ 重写（v0.5 留口） |

**关键不变量**：v0.6 已建好的 server 三连锁（id 共享 / L1 一致性 / S1-S7 优雅停机）保留不动；Phase 4 只做"加法"。

## 二、为什么不是 X

### 2.1 不是"OTel 用 Jaeger 直连不上 collector"

考虑过：Pod 内 OTel SDK 直发 Jaeger backend，省一层 collector。

**否决理由**：
- collector 是 trace 流水线的削峰组件，sidecar/sub-Pod 注入是 K8s 标准实践，简历叙事价值高
- 直发 Jaeger 在 Pod 数量增加时反压回 SDK 的 buffer 满 → 主路径阻塞风险
- collector 可统一加 sampling / batching / multi-backend 路由

### 2.2 不是"Ingress 上不做 AuthN，靠 NetworkPolicy 隔离"

考虑过：/api/stats/* 通过 NetworkPolicy 限制只 admin Pod 可访问。

**否决理由**：
- NetworkPolicy 是 L3/L4 IP/port 层，不是身份层
- admin 用户是人不是 Pod，需要可携带的凭证（token / basic auth / oauth）
- Ingress AuthN 是 L7 标准方案，与 K8s 解耦（换 ALB/Cloudflare 同样工作）
- NetworkPolicy 仍要做（防御纵深第二层），但不替代 AuthN

### 2.3 不是"consumer 仍留 host，等 v0.7 一起 K8s 化"

考虑过：v0.7 把 PG/Redis/Kafka/CH 全 K8s 化时，consumer 顺手一起进。

**否决理由**：
- v0.6 retro §3.1/§3.2 现场提示 consumer 单点是 v0.6 真隐患——仅次于 Kafka K8s advertised listener
- consumer K8s 化复杂度低（无状态 + Kafka group rebalance 自然支持多副本），可独立完成
- v0.7 真集群验 RPS 时如果 consumer 仍单点，等于 v0.7 演练里又埋一个 single point

### 2.4 不是"recon-fixture.sh 不重写靠 manual 对账"

v0.5 retro §3.7 已答：recon 时间窗算法 bug 让"R1 PASS/FAIL"误报，每次 drill 跑完看 round-*.csv tail 手工对比。

**否决理由**：
- v0.6 Day 27 演练盲点的根源就是 recon 不可信 → 工具不靠谱 → 强制对账步骤被跳过
- Phase 4 演练需要 consumer K8s 化后再演练一次，必须有可信的 recon 工具
- 时间窗算法重写工作量低（~30 行 bash），可复用现有 PG/CH SQL

## 三、模块图（v0.6 Phase 3 → Phase 4 收口）

### Phase 4 完成后

```
                                  [client]
                                      │
                                      ▼
                         ┌────────────────────────┐
                         │   Ingress (nginx)      │ ← Phase 4 (a)
                         │   - LB                 │
                         │   - AuthN（basic/oauth2-proxy） │
                         └────────────┬───────────┘
                                      │
                  ┌───────────────────┼───────────────────┐
                  ▼                   ▼                   ▼
          [server Pod 1]      [server Pod 2]      [server Pod 3]
          ├─ fasthttp         ├─ fasthttp         ├─ fasthttp
          ├─ L1 (Pod)         ├─ L1 (Pod)         ├─ L1 (Pod)
          ├─ id batch (Pod)   ├─ id batch         ├─ id batch
          └─ OTel exporter ───┴─ OTel exporter ───┴─ OTel exporter
                                      │
                                      ▼
                           ┌──────────────────┐
                           │ OTel Collector   │ ← Phase 4 (b) sidecar 或独立 Deployment
                           │ - batch          │
                           │ - sampling       │
                           │ - export Jaeger  │
                           └────────┬─────────┘
                                    ▼
                                 [Jaeger]
                                    │
                  ┌─────────────────┼─────────────────┐
                  ▼                 ▼                 ▼
              PG (host)         Redis (host)      Kafka (host)
              ├─ 号段表          └─ L2 cache        │
              └─ click_events                       │
                                ┌─────────────────┘
                                ▼
                  ┌──────────────────────────┐
                  │ consumer Deployment       │ ← Phase 4 (d) K8s 化 / replicas=2
                  │ replicas=2                │
                  │ Kafka group rebalance     │
                  │ S1-S7 优雅停机             │
                  └──────────────┬────────────┘
                                 │
                    ┌────────────┼────────────┐
                    ▼                         ▼
                  PG click_events    CH Engine + MV
                                          │
                                          ▼
                                    click_events_ch

NetworkPolicy ← Phase 4 (c)
- slink-server Pod → PG/Redis/Kafka 允许
- slink-consumer Pod → PG/Kafka 允许
- 任何其他 Pod → 后端 deny
- /api/stats/* 仅 ingress-controller 可访问
```

**关键变化**：
1. Ingress 取代 NodePort 成为入口（Phase 4 (a)）
2. OTel collector 独立 Deployment（Phase 4 (b)）
3. NetworkPolicy 默认 deny + 白名单（Phase 4 (c)）
4. consumer 进 Deployment 多副本（Phase 4 (d)）
5. recon-fixture.sh 重写（无图示，工具层）

## 六、灰度迁移路径（4 子阶段）

### Phase 4.1：OTel collector + trace 链路（Day 30）

- 决策稿 §7 (a)/(b) 拍板（spike 后）
- `internal/otel/` 包新建：trace exporter init + middleware + Kafka producer/consumer hook
- `deploy/k8s/30-otel-collector.yaml` Deployment + ConfigMap
- Pod sidecar 注入（如选 sidecar 形式）或环境变量指 collector
- 验证：单条业务请求在 Jaeger UI 能看到 client → Ingress → Pod → {PG, Redis, Kafka, CH} 全链路 span

**Phase 4.1 验收标准**：trace 链完整 + 主路径 P99 影响 < 5%（vs Phase 3 baseline）

### Phase 4.2：Ingress + AuthN（Day 31 上半段）

- 决策稿 §7 (c)/(d) 拍板（spike 后）
- `deploy/k8s/40-ingress-controller.yaml` 安装 ingress-nginx 或 traefik
- `deploy/k8s/41-ingress.yaml` slink Ingress 资源 + AuthN annotation
- `/api/links` / `GET /:code` 走 Ingress 业务路径
- `/api/stats/*` 走 Ingress + AuthN 鉴权路径
- 删除 NodePort（business path）+ 保留 admin NodePort 作 dev fallback

**Phase 4.2 验收标准**：业务路径不退步 + /api/stats/* 无 AuthN 拒绝（401）+ 携带凭证 200

### Phase 4.3：NetworkPolicy + consumer 入 K8s（Day 31 下半段）

- 决策稿 §7 (e) consumer K8s 化拍板
- `deploy/k8s/50-network-policy.yaml`：
  - 默认 deny all
  - slink-server → PG/Redis/Kafka 允许
  - slink-consumer → PG/Kafka 允许
  - ingress-controller → slink-server 允许
- `Dockerfile.consumer`（复用 server Dockerfile multi-stage）
- `deploy/k8s/60-consumer-deployment.yaml` replicas=2 + S1-S7 优雅停机配置
- host cmd/consumer 进程下线（保留 binary 作 dev fallback）

**Phase 4.3 验收标准**：NetworkPolicy 启用后业务不退步 + consumer 滚动期 0 漏（kafka group rebalance 自然处理）

### Phase 4.4：recon-fixture 重写 + Phase 4 演练 + 收口（Day 32）

- `scripts/recon-fixture.sh` v2 重写：
  - 时间窗算法基于灌数据期 max(ts)/min(ts) 而非 wall-clock now
  - 多 Pod 并发场景的多 partition 对账
  - R1/R2/R3 三档输出格式不变
- Phase 4 完整故障演练（仿 v0.6 Phase 3 三轮 + 加 consumer Pod kill）：
  - Round 1：`kubectl delete pod` server 单杀（v0.6 复跑）
  - Round 2：`scale 0/3` server 全杀（v0.6 复跑）
  - Round 3：`cordon` + `drain`（v0.6 复跑）
  - **Round 4（新）**：`kubectl delete pod` consumer 单杀
  - **Round 5（新）**：`scale 0/2` consumer 全杀 + Kafka group rebalance 验证
- v0.6 Phase 4 收口：tag `v0.6-final`（覆盖 `v0.6-k8s-final` 或并存）+ retro 增量章节

**Phase 4.4 验收标准**：5 轮演练全 0 timeout + 端到端 drift 5× 0.0000 + recon-fixture 报告与 manual 对账数字一致

**预估全程**：3-4 work day（Day 30-33，可单日多段打穿沿用 v0.6 节奏）。
