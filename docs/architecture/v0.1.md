# slink v0.1 架构

> 状态：🚧 开发中 · 目标：单机 5w+ QPS · 范围：API + 跳转 + 缓存 + 异步事件

## 一、能力边界

v0.1 **能做**：

- ✅ 创建短链：`POST /api/links` → 生成 6-7 位 Base62 短码
- ✅ 跳转：`GET /:code` → 302 重定向 + 异步记录点击事件
- ✅ 单机 Redis 缓存（基础穿透/雪崩防护）
- ✅ 异步事件入库（channel + batch）
- ✅ Docker 单机部署
- ✅ wrk 压测可达 5w+ QPS

v0.1 **不做**（留给 v0.2/v0.3）：

- ❌ Kafka 削峰
- ❌ 多级缓存（本地 LRU + 布隆过滤器）
- ❌ 实时聚合（HLL UV、TopK）
- ❌ 分布式部署 / 选主
- ❌ ES 历史检索 / PG 分区归档

## 三、数据流

### 创建短链（低 QPS）

```
client
  │ POST /api/links {long_url}
  ▼
api.CreateLink
  │ 1. 校验 long_url 格式
  │ 2. 检查 idempotency key（如带）
  ▼
id.Segment.NextID()
  │ 内存号段未耗尽 → 直接 +1
  │ 已耗尽 → UPDATE id_segment 取下一段
  ▼
id.Base62.Encode(id) → "5BxX"
  │
  ▼
store.Links.Insert(id, code, long_url)
  │
  ▼
cache.Set("link:5BxX", long_url, ttl)
  │
  ▼
return {code: "5BxX", short_url: "..."}
```

### 跳转（高 QPS，10w 目标）

```
client (浏览器/短信跳转)
  │ GET /5BxX
  ▼
api.Resolve
  │
  ▼
cache.Get("link:5BxX")
  ├─ HIT  → long_url
  └─ MISS → store.Links.GetByCode("5BxX") → long_url → cache.Set
  │
  ▼
http.Redirect(302, long_url)         ◄─── 用户立即跳走
  │
  ▼ (异步，不阻塞响应)
event.Buffer.Enqueue(ClickEvent{...})
  │
  ▼ (后台 goroutine 每秒一刷或满 1000 条一刷)
store.ClickEvents.BatchInsert([]ClickEvent)
```

## 四、关键决策概览

| 决策 | 选了 | 没选 | 理由 | 详档 |
|---|---|---|---|---|
| 主存储 | PostgreSQL 16 | MySQL 8 | 原生分区、JSONB、partial index、CTE | [ADR-0001](../adr/0001-postgres-as-primary-store.md) |
| ID 生成 | 号段模式 | Snowflake / UUID / Hash | 短码长度敏感、可控空间 | [ADR-0002](../adr/0002-id-segment-not-snowflake.md) |
| 事件表 | 按月 RANGE 分区 | 单表 + 索引 | 海量数据归档（DROP PARTITION）瞬间完成 | [postgres-partitioning](../concepts/postgres-partitioning.md) |
| 缓存 | Redis（v0.1 仅一级） | 仅 PG / 全本地 | v0.1 起步，v0.2 加多级 | (v0.2 文档) |
| 重定向 | HTTP 302 | 301 | 每次跳转打到服务端，**保事件采集** | (Day 5 文档) |
| 项目布局 | cmd + internal | flat / pkg/ | 清晰的私有边界 | [go-project-layout](../concepts/go-project-layout.md) |

## 七、当前限制（必须诚实承认）

1. **单机部署**：服务挂了就挂了，没有 HA。v0.2/v0.3 加分布式
2. **Redis 是 SPOF**：Redis 挂 = 所有跳转打 PG = PG 大概率挂。v0.2 加本地缓存兜底
3. **进程崩溃丢事件**：buffer 在内存，崩了那一秒的事件丢失。v0.2 切 Kafka
4. **号段重启浪费**：每次重启浪费当前未用完的号段（影响极小，号段空间几百亿）
5. **分区只预建到 2026-06**：需要后台任务自动建下个月分区，v0.1 暂手动

## 八、与 v0.2 / v0.3 的衔接点

v0.1 设计时已为后续版本留接口：

| v0.1 实现 | v0.2/v0.3 升级路径 |
|---|---|
| `event.Buffer` 是 channel + batch | 替换为 Kafka producer，对外 API 不变 |
| `cache.Cache` 是 Redis 直连接口 | 包一层 multi-tier，先查本地再查 Redis |
| `id.Generator` 接口 | 多机部署时切到分布式号段（DB 行锁/CAS） |
| `store.ClickEvents` 已分区 | 加月度建分区任务即可 |

接口隔离保证 v0.1 → v0.2 是**替换实现**而非**重构架构**。
