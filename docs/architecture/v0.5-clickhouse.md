## 一、为什么 v0.5 是 ClickHouse

### 1.1 v0.4 留的真问题

v0.4 收口（Day 16 切流）后链路：

```
GET /:code ──► fasthttp ──► KafkaProducer ──► Kafka topic
                                                  │
                                                  ▼
                                            PG Consumer ──► click_events 表（PG 行存）
```

事件落了 PG，能查"某个 code 被点了几次"——按 (code, ts) 索引扫一下就行。

但回答不了：
- "过去 1 小时 UV 是多少"——COUNT(DISTINCT ip) 在 PG 行存上扫月分区是分钟级
- "TopK 热门 code"——GROUP BY code ORDER BY count DESC 同样扫表
- "按 region/UA 维度的下钻"——临时聚合 OLTP 撑不住

**根因结构性**：PG 是 OLTP，行存 + 索引适合"查某一行"，不适合"扫一段时间的聚合"。click_events 现有 schema（按月分区 + (code,ts) 索引）已经把 PG 榨到极限。

### 1.2 v0.5 目标

| 维度 | v0.4 | v0.5 目标 |
|---|---|---|
| 主路径不动 | ✅ | ✅（绝对不变）|
| Kafka 路径捕获率 | 97.9% | 不退步 |
| PG click_events 落库 | 0 漏 | 不退步（保留作为审计源）|
| **新增：实时 UV/TopK 查询延迟** | n/a（PG 算分钟级）| **< 200ms（ClickHouse + HLL）** |
| **新增：UV 聚合数据保留期** | n/a | 90 天（按需调）|
| **新增：故障域** | 2（Kafka / PG）| 3（Kafka / PG / ClickHouse 互不影响）|

**关键不变量**：v0.4 已建好的"主路径绝不为下游退步"原则继续。ClickHouse 抖动只影响"分析查询"，不影响"事件落 PG"。

## 二、为什么不是 X

### 2.1 不是"PG 物化视图 / 增量聚合"

考虑过：在 PG 上建 materialized view + cron 刷新，或者用 trigger 维护聚合表。

**否决理由：**
- 仍然在 PG 同实例上挤资源——主路径 click_events 写入和分析查询竞争连接池 / cache / WAL
- 物化视图刷新本身是大查询，频率低就实时性差，频率高就抢资源
- HLL UV / TopK 在 PG 上没有原生算子（pg_stat_statements 是统计 SQL 本身，不是数据维度）

### 2.2 不是"K8s + OpenTelemetry"

v0.4 复盘 §4 已论证：**v0.5 = ClickHouse 是 v0.4 决策稿明确预留的扩展点**（"下游可扩展：v0.5 加 HLL UV 时多挂一个 consumer 就行"，§3）。

K8s 不是不做，而是 **v0.6 做**——因为：
- ClickHouse 加 consumer 是顺势，pipeline 现成
- K8s 多副本会触发 server 无状态化连锁问题（id 号段共享 / L1 一致性 / Pod 滚动），是改核心架构
- 一次只动一个维度（v0.5 只动分析侧 / v0.6 只动部署侧）

### 2.3 不是"ElasticSearch / Druid / Pinot"

考虑过：ES 是日志分析常见栈；Druid/Pinot 是 LinkedIn/Uber 的 OLAP 选型。

**否决理由：**
- ES 是文档检索引擎，硬塞分析能用但 OLAP 性能不如 ClickHouse 1 个量级
- Druid/Pinot 学习曲线 + 部署复杂度高出 ClickHouse 2-3 倍，单机项目不值
- ClickHouse 单机起步即可，K8s 上线时再考虑 sharding；学习曲线和 PG 接近

### 2.4 不是"TimescaleDB"

考虑过：TimescaleDB = PG + 时间序列扩展，可以复用现有 PG。

**否决理由：**
- 仍在 PG 同实例上跑（同 2.1）
- 时间维度优化但 cardinality 高的维度（按 code/region/UA 聚合）优化有限
- HLL 不是原生算子，需要插件
- 真要解决"分析侧无能"，列存比行存优势是 1-2 个量级，TimescaleDB 仍是行存

### 2.5 为什么是 ClickHouse

- **列存天生 OLAP**：扫聚合查询比 PG 快 1-2 个量级
- **HLL 原生算子**：`uniqHLL12(ip)` 一行 SQL 拿到近似 UV，0.81% 标准误
- **Kafka engine 表**：ClickHouse 自带"从 Kafka 消费 + 写本地表"语法，consumer 代码量小
- **MergeTree 引擎**：写多读少 + 时间分区天然契合 click 流
- **业界标准**：Cloudflare / Uber / Yandex 都用，简历叙事可对线
- **v0.4 决策稿明确预留**：架构稿 §3 / §6 / §10 都标注"v0.5 加 ClickHouse consumer"

## 三、模块图（v0.4 → v0.5）

### v0.4 现状（Day 16 切流后）

```
GET /:code ──► fasthttp ──► KafkaProducer ──► Kafka (4 partition)
                                                  │
                                                  ▼
                                            PG Consumer
                                                  │
                                                  ▼
                                          click_events (PG 行存)
                                              │
                                              ▼
                                          慢分析查询（COUNT(DISTINCT ip) 等）
```

### v0.5 目标

```
GET /:code ──► fasthttp ──► KafkaProducer ──► Kafka (4 partition)
                                                  │
                              ┌───────────────────┼───────────────────┐
                              ▼                                       ▼
                        PG Consumer                          ClickHouse Consumer (v0.5 新增)
                        (group=pg_writer)                    (group=clickhouse_writer)
                              │                                       │
                              ▼                                       ▼
                        click_events                          click_events_ch (CH MergeTree)
                        (PG 行存，审计源)                       (CH 列存，分析源)
                                                                      │
                                                                      ▼
                                                              新 admin endpoint：
                                                              /api/stats/uv?code=X&from=Y&to=Z
                                                              /api/stats/topk?from=Y&to=Z&n=10
                                                              ...
```

**关键设计**：两个 consumer group 独立 commit，互不影响。CH 挂了，PG 路径仍正常落库。

## 四、待 spike 的关键不确定项

v0.4 立项时（Day 12-13）做过 sarama vs kgo spike，结果决定主线。v0.5 同样需要先 spike：

| 不确定项 | spike 内容 | 决定什么 |
|---|---|---|
| **ClickHouse 客户端库** | clickhouse-go v2 vs ch-go vs JDBC 等 | consumer 写 CH 用哪个库 |
| **写入模式** | INSERT VALUES vs Native Protocol vs Kafka Engine 直消 | Kafka → CH 三种入数据姿势的 throughput / 复杂度 |
| **HLL 精度 vs 内存** | uniqHLL12 vs uniqCombined vs uniqExact 对比 | 准确度和内存占用 trade-off |
| **batch 参数** | CH 推荐 ≥ 100ms / ≥ 1000 行批量；与 v0.4 consumer.BatchTimeout=100ms 一致还是单独调 | consumer batch 节奏 |
| **schema** | (event_id, code, ip, ts, ...) 列序 / partition by toYYYYMM(ts) / order by (code, ts) | CH 表 DDL |

**spike 输出标准**（参照 Day 13 sarama-vs-kgo 同口径方法）：
- 同硬件 / 同 Kafka topic / 30s pubsub
- 测 throughput（records/s）+ alloc/op + 部署复杂度
- 决策稿封板，禁止后续二次选型

## 五、范围红线（v0.5 不做）

明确不做的事，避免 scope creep：

- ❌ **替换 PG 路径** — PG click_events 留作审计源 + 数据回溯，不删
- ❌ **多 ClickHouse 副本** — 单机起步，集群（sharding/replication）留 v0.6+
- ❌ **K8s 部署** — v0.6 主战场
- ❌ **OpenTelemetry trace** — v0.6+
- ❌ **改 producer / 主路径** — 任何 producer 改动都需要回 v0.4 架构稿对齐
- ❌ **Schema Registry / Proto** — A3 加的 V 字段是 JSON 内 schema 演化的过渡方案，proto 切换留 v0.7+
- ❌ **新业务功能**（如 link expiration / batch redirect API）— v0.5 范围只到"分析侧"

任何"v0.5 顺手做一下..."的诱惑都先入待办表，不做。

## 七、决策记录（Day 17 kickoff）

按本架构稿落地的 v0.5 关键决策（kickoff 时已能拍板的项）：

| 决策 | 选项 | 选了 | 理由 |
|---|---|---|---|
| 主线方向 | ClickHouse / K8s+OTel / Schema Registry | **ClickHouse** | v0.4 决策稿 §3 明确预留点，pipeline 现成 |
| 数据流向 | 替换 PG / 并行（PG + CH） | **并行** | PG 留作审计源，CH 做分析源 |
| Consumer 模型 | 共享 group / 独立 group | **独立 group** | 故障域分离（v0.4 验证过的设计原则）|
| group 命名 | clickhouse_writer / ch_writer / analytics | **slink.click_events.clickhouse_writer** | v0.4 命名前缀习惯 |
| UV 算法 | 精确 COUNT(DISTINCT) / HLL / SuperMinHash | **HLL（uniqHLL12）** | 0.81% 误差换 1-2 个量级内存 |
| 表引擎 | MergeTree / ReplacingMergeTree / SummingMergeTree | **MergeTree** | 起步最简，去重已在 producer/PG 侧做（v0.4 at-least-once） |
| 时间分区 | by day / by month | **by month**（toYYYYMM(ts)）| 同 PG click_events 分区策略，对账方便 |
| order by | (code, ts) / (ts, code) | **(code, ts)** | 同 PG 索引，主查询模式按 code 聚合 |

待 spike 后封板的项（§4 列表）：CH 客户端库 / 写入模式 / batch 参数 / schema 细节。

## 八、复用 v0.4 的工程习惯

v0.5 严格沿用 v0.4 验证过的三件套：

1. **决策稿先行**（本文）：所有关键决策落稿，禁止"边做边改"
2. **feature flag**：`SLINK_ANALYTICS_BACKEND=off|clickhouse` 控开关，方便回退
3. **同口径 spike**：CH 库选型必须 30s 同硬件 bench
4. **故障演练**：CH 故障 15s 注入，主路径 + PG 路径不退步是硬指标
5. **git tag**：v0.5 切流前打 `v0.4-final` 锚定可回滚 commit
6. **schema 演化**：CH 表 DDL 改动遵循 A3 同样原则（加列 ok / 改类型必升 schema_version）
